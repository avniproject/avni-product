### Primary purpose

Fully custom cards allow organisations to display dynamic, HTML-rendered views of data on the Avni Android dashboard. Organisations such as AKRSP need field users to be able to share these rendered card views — for example, to send a summary of forms filled in a custom format via WhatsApp or email. This card adds share capability to the fully custom card type.

Epic: [avni-product#1849 – Custom design cards](https://github.com/avniproject/avni-product/issues/1849)
Card: [avni-client#1904 – [Fully custom card] Ability to share](https://github.com/avniproject/avni-client/issues/1904)

---

### Current state

Fully custom cards are being introduced as part of the Custom Design Cards epic. The card renders an HTML file uploaded by the implementer, populated with data returned by a JavaScript rule. There is currently no share functionality on any card type in the Avni Android app.

---

### The feature

A share action is available on fully custom cards, allowing field users to share the rendered card view as a PDF via the Android native share sheet.

**Acceptance Criteria**

- The standard Android share icon (SVG: three circles connected by lines) is placed in the Avni app toolbar when viewing a fully custom card — it is completely outside the implementer-rendered HTML area
- Tapping the share icon converts the current rendered state of the custom card (HTML + populated data) to a PDF
- The native Android share sheet is invoked with the PDF attached, allowing the user to share via WhatsApp, Gmail, Google Drive, or any other installed app
- The PDF reflects the card exactly as rendered on screen, including any active filters applied (e.g. a month filter selected by the user)
- The shared PDF includes the full content of the card — if the HTML is taller than the screen (scrollable), the entire content is captured across multiple pages in the PDF (not just the visible portion)
- The PDF is named after the report card (e.g. `payment_dec_2025.pdf` for a card named "Payment Dec 2025")
- The PDF includes a watermark at the bottom showing: `Generated by Avni · <user name> · <date and time>` (e.g. `Generated by Avni · Raju Kumar · 20-04-2025 14:32`)
- The share icon is hidden or disabled when the card is in a loading state or has encountered an error

---

### Out of scope

- Sharing from DEA (explicitly excluded in epic)
- Print functionality (excluded in epic)
- Sharing individual form fills or raw data — only the rendered card view is shared
- Sharing from standard report cards or custom query cards (only fully custom card type)
- iOS support

### Technical details

**PDF generation approach**

The recommended approach is to use a library such as [`react-native-print`](https://github.com/christopherdro/react-native-print) or [`react-native-html-to-pdf`](https://github.com/christopherdro/react-native-html-to-pdf) to convert the custom card's HTML content to a PDF file before invoking the Android share intent.

**Scrollable content and automatic pagination**

Libraries like `react-native-html-to-pdf` generate the PDF by rendering the full HTML document (not just the visible viewport) and automatically paginating it. Content that overflows a single page is split across multiple pages using the browser's print layout algorithm. The implementer's HTML does not need any special changes — standard CSS `@media print` rules apply. This means:

- If the custom card shows 30 rows of data and the user can scroll, all 30 rows will appear in the PDF, spread across however many pages are needed
- Page breaks follow CSS rules (`page-break-before`, `page-break-after`, `page-break-inside`) if specified by the implementer; otherwise content reflows naturally
- The `react-native-print` library similarly renders the full HTML and hands off to the Android print subsystem, which handles pagination

**PDF filename**

The PDF should be named after the report card. For example, if the card is named "Payment Details", the file should be `payment_details.pdf`. If the card supports a date/month filter, the active filter value can be appended (e.g. `payment_details_dec_2025.pdf`). The exact naming convention to be confirmed during implementation.

**Watermark**

A watermark line is added at the bottom of the PDF containing: `Generated by Avni · <logged-in user name> · <date and time of generation>` (e.g. `Generated by Avni · Raju Kumar · 20-04-2025 14:32`). This can be implemented as a fixed footer injected into the HTML before conversion. The logged-in user's name and the generation timestamp are available at the time the share action is triggered.

**Share icon placement**

The share icon is rendered inside the Avni app toolbar (outside the WebView that renders the implementer's HTML). This ensures the implementer's HTML card is completely untouched and the share action is a platform-level concern.

**Invoking the share sheet**

Once the PDF file is generated and saved to a temporary location, Android `ShareCompat.IntentBuilder` (or the React Native bridge equivalent) is used to invoke the native OS share sheet with the PDF as the shared attachment.
