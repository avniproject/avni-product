### Primary purpose

AKRSP needs to generate PDFs of form fills in a government-mandated format — not Avni's default table layout — so field workers can produce documents that match official registers. This requires a configurable "Share rule" that implementers set up per form in the web admin, defining the custom output format and translated labels.

This feature builds on top of the basic form-share-pdf feature (share icon on each form fill row) but replaces the default Avni format with a custom implementer-configured layout when a Share rule is present.

---

### Current state

The form-share-pdf feature (basic) generates a fixed Avni format PDF: subject header → form metadata → section headers → field/value table. There is no way to customise the layout, field labels, language, or field selection per form.

---

### The feature

**Share rule** is a per-form configuration in the web admin that defines:
- Which fields to include in the PDF
- Custom labels for each field (overriding the Avni field label), with translations per language
- Field ordering and grouping
- Output layout (initially: table format matching government register)

When a Share rule is configured for a form and the user taps "Share" on a fill of that form, the PDF is generated using the Share rule's format instead of the default Avni format. When no Share rule is configured, the default format is used.

---

### Input format — Share rule configuration

The Share rule is configured in the web admin per form. It consists of:

**Rule header**

| Property | Description |
|---|---|
| Form | The form this rule applies to |
| Output format | Layout type — initially only "Table" (matching government register) |
| Language | Default language for labels in the PDF |

**Field mapping rows** (one per field to include)

| Property | Description |
|---|---|
| Avni field | The source field from the form |
| Include | Whether this field appears in the PDF |
| Display order | Sequence in which this field appears |
| Section / group | Optional grouping header above this field in the PDF |
| Label (default) | Custom label to use in the PDF (overrides Avni field label) |
| Label translations | Label translated per configured language (e.g. Marathi, Hindi) |

**Stored as JSON on the server**, associated with the form. Example structure:

```json
{
  "formUUID": "...",
  "outputFormat": "TABLE",
  "fields": [
    {
      "conceptUUID": "...",
      "include": true,
      "order": 1,
      "section": "Details",
      "labels": {
        "en": "Total farm area (silt spread)",
        "mr": "एकूण शेत क्षेत्र (गाळ पसरलेले)"
      }
    },
    {
      "conceptUUID": "...",
      "include": false
    }
  ]
}
```

Fields with `"include": false` are excluded from the PDF. Media fields (photos, audio, video) are always excluded regardless of configuration.

---

### Return format — PDF output

When Share is tapped on a form fill that has a Share rule configured:

1. **Subject header** (always present): subject name + demographic line (name, gender, age, address hierarchy) — same as default format
2. **Form metadata** (always present): form type name, visit date, filled by, visit type
3. **Body**: fields rendered in the order defined by the Share rule, grouped under section headers if configured, with custom translated labels
4. **Watermark** (always present): `Generated by Avni · <user name> · <date time>`

The PDF is generated from HTML using `react-native-html-to-pdf`. The HTML template is assembled at share time by merging the form fill data with the Share rule configuration.

**Language selection**: the language used for labels is determined by the language configured on the device (or the user's language preference in Avni). If no translation exists for the device language, the default label falls back to English.

**Filename**: derived from form type + date, same as default format (e.g. `farmer_endline_04_02_2026.pdf`).

---

### Translation export / import

- In the webapp, after configuring field labels for a form's Share rule, the implementer can **export** the label mapping as a CSV/Excel file
- The export includes one row per field, with columns for each configured language
- This allows offline translation work and bulk import back into the webapp

Export format (CSV):

```
field_uuid, field_name_en, label_en, label_mr, label_hi
abc-123, Total farm area, Total farm area (silt spread), एकूण शेत क्षेत्र (गाळ पसरलेले), कुल कृषि क्षेत्र
```

---

### Behaviour summary

| Scenario | Behaviour |
|---|---|
| Share rule configured, device language has translation | PDF generated with custom labels in device language |
| Share rule configured, no translation for device language | PDF generated with custom labels in default (English) |
| No Share rule configured for the form | PDF generated in default Avni format |
| Field marked `include: false` in Share rule | Field omitted from PDF |
| Media field | Always omitted from PDF, regardless of Share rule |

### Out of scope

- iOS support
- Custom PDF layout beyond table format (e.g. pre-printed form overlay, letterhead)
- Sharing blank/empty form templates
- Bulk sharing (multiple form fills at once)
- Share rules for Registration forms vs Encounter forms — to be confirmed (see open questions)

### Technical details

**Web admin (avni-webapp)**

- New "Share rule" section in the form configuration screen
- Field mapping table: drag-to-reorder, include/exclude toggle, label input per language
- Export translations as CSV
- Import translations from CSV
- Share rule stored as part of form configuration, serialised to JSON

**Server (avni-server)**

- New entity: `FormShareRule` — associated with a `Form`, stores field order, include flags, and label translations per concept per language
- Exposed via existing form config API endpoint (or new dedicated endpoint)
- Synced to mobile as part of form metadata sync

**Mobile app (avni-client)**

- On sync, store `FormShareRule` locally alongside form metadata
- On "Share" tap: check if a Share rule exists for this form
  - If yes: use Share rule to assemble HTML (custom labels, ordering, field selection)
  - If no: use default Avni format
- Determine device/user language → look up label translation → fall back to English if missing
- Generate PDF from assembled HTML → invoke Android share sheet

**Affected repositories**

| Repository | Change |
|---|---|
| `avni-webapp` | Share rule UI in form config; translation export/import |
| `avni-server` | `FormShareRule` entity; storage and sync API |
| `avni-client` | Share rule lookup; HTML assembly with custom labels; language fallback |

### Open questions

1. **Image in AC not accessible**: The AC includes a screenshot of the government register format that could not be loaded. This needs to be reviewed — it likely defines the exact PDF layout required and may reveal additional requirements (e.g. pre-printed form overlay, specific column widths, logo, etc.).
2. **Registration vs Encounter forms**: Does the Share rule apply to Registration form fills as well, or only Encounters?
3. **Multiple Share rules per form**: Can a form have different Share rules for different organisations, or is it one rule per form globally?
4. **Section headers**: Are section headers in the Share rule free-text (implementer-defined), or do they map to the form's existing section groupings?
5. **Default language**: Where is the default PDF language configured — at org level, user level, or per Share rule?
6. **Translation import**: Is bulk import of translations (CSV back into webapp) in scope for the initial release?
7. **Sync strategy**: Is the Share rule synced as part of the existing form metadata sync, or does it need a separate sync endpoint?
